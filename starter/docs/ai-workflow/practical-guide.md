# Practical Guide

このドキュメントは、AI駆動開発で1つのタスクをどう進めるかを説明します。

テンプレートを埋めることが目的ではありません。
目的は、AIを使っても作業範囲、判断理由、検証結果、残課題が追える状態にすることです。

## 1. 全体計画と今回のタスクを分ける

プロジェクトを始めるときは、まず作りたいもの、目的、制約、成功条件を整理し、`docs/ROADMAP.md` に全体計画を書きます。
新しいセッションでは、`docs/PROJECT_STATUS.md` と `docs/ROADMAP.md` を見て現在の開発状況を把握し、今回取り組むタスクを1つに絞ります。

役割:

- `docs/ROADMAP.md`: プロジェクト全体の地図
- `docs/PROJECT_STATUS.md`: 現在地と次の候補タスク
- `docs/DIRECTORY_MAP.md`: 主要ディレクトリ、責務、タスク別の参照先
- `docs/tasks/.../implementation-plan.md`: 今回の1タスクの実行計画

ROADMAPの全項目を一度に実装しようとしないことが重要です。
AI支援開発では実装速度が速いため、大きな目的を小さなタスクに分けてから進めます。

## 2. 要件認識合わせを行う

AIに実装を頼む前に、まずタスクを明確にします。

確認すること:

- ユーザーの元要望は何か
- AI側はどう理解したか
- 何を達成したいか
- なぜ今やるのか
- どの画面、API、ファイル、データが関係しそうか
- 何は今回やらないのか
- 何ができたら完了と言えるか
- どう検証するか
- AIが勝手に置いた仮定はないか
- ユーザーに確認すべき不明点はないか

曖昧なままAIに渡すと、AIは足りない情報を推測します。
その推測が外れると、見た目は進んでいても、違うものを作っている状態になります。

必要なら `templates/requirement-alignment.md` と `templates/task-brief.md` を使います。
作成した記録は、テンプレート本体ではなく `docs/tasks/YYYY-MM-DD-HHMM-task-name/` に保存します。
未確認の前提が残る場合は、実装に進みません。

## 3. 現在の開発状況を把握する

新しいチャットや新しいAIツールで作業を始めるときは、最初に現在の開発状況を把握します。

確認すること:

- `git status --short --branch`
- `README.md`
- `AGENTS.md`
- `docs/PROJECT_STATUS.md`
- `docs/ROADMAP.md`
- `docs/DIRECTORY_MAP.md`
- 直近の `docs/devlog/`
- 最後に通ったテストやビルド

ここを省くと、AIは古い前提や不足した文脈で作業します。
結果として、既存設計に合わない実装や、前回見送った方針を再実行する危険が増えます。

手順は `workflows/session-start.md` を使います。

## 4. Directory Contextを確認する

今回のタスクが決まったら、実装前に `docs/DIRECTORY_MAP.md` を読みます。
目的はコード全体を読むことではなく、今回のタスクで読むべき範囲を絞ることです。

確認すること:

- 今回のタスクはどのディレクトリに関係するか
- そのディレクトリの責務は何か
- 関連して読むべきディレクトリはどこか
- 触ってはいけない境界はどこか
- DIRECTORY_MAPと実際の構成に大きなズレがないか

DIRECTORY_MAPがない場合は、実装前に主要ディレクトリと責務を短く整理します。
新しい主要ディレクトリや責務変更が必要になりそうな場合は、implementation planに更新候補として記録します。

## 5. Workflow Modeを判定する

今回のタスクが決まったら、AIがWorkflow Modeを判定します。
導入時にユーザーがMinimal、Standard、Strictを固定する必要はありません。

デフォルトはStandardです。
小さく戻しやすい低リスク作業だけMinimalに下げ、認証、権限、課金、個人情報、本番設定、外部依存、公開リリース、アーキテクチャ変更を含む場合はStrictに上げます。

判定結果と理由は、implementation planに記録します。
Strictの可能性があるが判断できない場合は、実装前にユーザーへ確認します。

## 6. 実装計画を作る

実装前に、計画を短く書きます。

計画で固定すること:

- 目的
- 背景
- 変更するもの
- 変更しないもの
- Directory Context
- 実装方針
- 影響範囲
- リスク
- Workflow Modeと判定理由
- 検証方法
- 完了条件

計画は長文である必要はありません。
重要なのは、AIが勝手に範囲を広げたときに「それは今回の対象外」と判断できる基準を持つことです。

テンプレートは `templates/implementation-plan.md` を使います。
保存先は `docs/tasks/YYYY-MM-DD-HHMM-task-name/implementation-plan.md` です。

## 7. AIに小さく任せる

AIには大きなタスクを一度に投げない方が安全です。

良い任せ方:

- 関連ファイルを調べる
- 実装案を比較する
- 小さな差分を作る
- テストケースを洗い出す
- エラー原因の候補を出す
- ドキュメントの下書きを作る

危険な任せ方:

- 仕様が曖昧なまま実装させる
- ユーザーに確認すべき不明点をAIが勝手に補完する
- 大量のファイルを一括で変更させる
- 認証、権限、課金、個人情報をAI判断だけで変える
- テストせずに完了扱いにする
- AIの説明だけを根拠にマージする

AIの出力は、完成品ではなくレビュー対象の提案です。

## 8. 差分を確認する

実装後は、コードが動くかだけでなく、差分が計画に合っているかを確認します。

確認すること:

- 計画した範囲だけが変わっているか
- 意図していないファイルが変わっていないか
- 既存の設計や命名に沿っているか
- 不要なリファクタリングが混ざっていないか
- エラー処理、境界値、権限、データ扱いに問題がないか

予定外の変更が必要になった場合は、隠さず記録します。

## 9. 検証する

完了判断は、気分ではなく証拠で行います。

例:

```bash
npm test
npm run build
git diff --check
```

UIや操作が関係する場合は、手動確認も記録します。

記録する内容:

- 実行したコマンド
- 結果
- 失敗した場合の原因
- スキップしたチェックと理由

「AIが正しいと言った」は検証ではありません。

## 10. 完了前レビューを行う

完了前に、計画と実際の結果を照合します。

確認すること:

- 目的は達成されたか
- 完了条件を満たしたか
- 未完了のことは何か
- 計画から変わったことは何か
- 検証結果は十分か
- DIRECTORY_MAPの更新要否を確認したか
- 次の作業者が再開できるか

テンプレートは `templates/completion-review.md` を使います。
保存先は `docs/tasks/YYYY-MM-DD-HHMM-task-name/completion-review.md` です。
完了判断の基準は `docs/ai-workflow/definition-of-done.md` を使います。

## 11. 判断理由を残す

最後にdevlogを書きます。

devlogに残すこと:

- なぜこの作業をしたか
- 何を変更したか
- どの方針を採用したか
- どの選択肢を見送ったか
- 何を検証したか
- 何が未完了か
- DIRECTORY_MAPを更新したか、または更新不要な理由
- 次に何を見るべきか

devlogは作業日報ではありません。
将来の自分や他の開発者が、判断を追うための記録です。

テンプレートは `templates/devlog.md` を使います。

1タスク1セッションで進める場合、タスクを実行したら必ずdevlogを残します。
小さい修正では短くて構いませんが、判断理由、検証結果、未完了事項の3つは省略しません。

保存先の例:

```text
docs/devlog/YYYY-MM-DD/HHMM-task-name.md
```

タスク記録の保存先は [task-records.md](task-records.md) を参照してください。

## 12. 重要な設計判断はADRに分ける

devlogはセッション単位の記録です。
長期的に残すべき設計判断はADRに分けます。

ADRに残すべき例:

- 認証方式を変える
- 状態管理ライブラリを採用する
- DB構造を変える
- 外部サービスを導入する
- アーキテクチャを変更する

AIはADRを勝手に確定しません。
ADR候補を検知した場合は、実装前または完了前レビューでユーザーに確認します。
ユーザーが了承した場合のみ、ADRを作成します。

ADR候補の判断基準は [adr-guidelines.md](adr-guidelines.md) を参照してください。
テンプレートは `templates/adr.md` を使います。

## 13. PRとCIに接続する

GitHubで開発する場合は、PRに次を残します。

- 何を変えたか
- なぜ変えたか
- どのモードで作業したか
- 関連する計画、レビュー、devlog、ADR
- 実行した検証
- 残るリスク

`.github/pull_request_template.md` を使うと、レビュー時に確認しやすくなります。

必要に応じて、プロジェクト側のCIで文書構造、リンク切れ、秘密情報らしき文字列を確認します。

## まとめ

AI駆動開発で重要なのは、AIを使うこと自体ではありません。

重要なのは、AIで速くなった作業を、人間が検証できる形に戻すことです。

そのために、このガイドでは次を残します。

- 目的
- 範囲
- 判断理由
- 検証結果
- 未完了事項
- 次の再開地点
