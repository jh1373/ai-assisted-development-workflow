# Requirement Alignment First

AI駆動開発では、実装速度よりも先に、ユーザーとAIの認識を揃えます。

ユーザーが「こういう機能を作りたい」「ここを直したい」と短く伝えた場合、AIはすぐに実装へ進んではいけません。
まず、何を作るのか、何を作らないのか、どこまでできたら完了なのかを確認します。

このリポジトリでは、この考え方を **Requirement Alignment First**、日本語では **要件認識合わせファースト** と呼びます。

PROJECT_BRIEFは新規プロジェクト全体の目的、仮説、不明点を初期設定で整理します。
requirement alignmentは、その基準線の中で今回の1タスクについて認識を揃えます。
PROJECT_BRIEFがあることを理由に、タスクごとの認識合わせを省略してはいけません。

## なぜ必要か

AIは、曖昧な依頼でも足りない情報を推測して実装できます。
これは便利ですが、プロダクト開発では大きなリスクです。

認識がズレたまま実装が進むと、次の問題が起きます。

- ユーザーが欲しかった機能と違うものができる
- 本来やらない範囲までAIが変更する
- UX、データ、権限、課金、公開範囲の前提がズレる
- 後からの修正コストが大きくなる
- どの時点で認識がズレたのか追えなくなる

そのため、実装前に認識合わせを行います。

## 基本ルール

AIは、次が曖昧な場合、実装に進みません。

- ユーザーの目的
- 実装する範囲
- 実装しない範囲
- 完了条件
- 制約
- UXや挙動
- データの扱い
- 認証、権限、課金、個人情報への影響
- 公開範囲やリリース影響

曖昧な点がある場合は、AIが勝手に補完せず、ユーザーに確認します。

## 実装前に行うこと

AIは、実装前に次を整理します。

1. ユーザーの元要望を要約する
2. AI側の理解を明示する
3. 実装対象と対象外を分ける
4. 不明点、曖昧な点、懸念点を列挙する
5. AIが置いた仮定を明示する
6. ユーザーに確認すべき質問を出す
7. ユーザー確認が必要な場合は、回答を待つ
8. 合意できた範囲だけをタスク化する

## 実装に進んでよい状態

実装に進んでよいのは、次の状態です。

- ユーザーの目的をAIが説明できる
- 実装する範囲としない範囲が明確である
- 完了条件が明確である
- 重要な不明点が残っていない
- AIが置いた仮定がユーザーに見えている
- 必要なユーザー確認が終わっている

未回答の不明点がある場合は、実装に進みません。

## 例外

低リスクで意図が明確な小修正では、長い確認は不要です。

例:

- 誤字修正
- 明らかなリンク切れ修正
- コメントの軽微な修正
- テスト名の軽微な修正

ただし、小さい変更でも、挙動、UX、データ、セキュリティ、コスト、公開範囲に影響する場合は確認します。

## 悪い進め方

```text
User: 検索機能をいい感じにして
AI: 実装しました
```

問題:

- 「いい感じ」の意味が不明
- UI変更なのか、検索ロジック変更なのか、API変更なのか不明
- 完了条件が不明
- ユーザーが望まない変更が混ざる可能性が高い

## 良い進め方

```text
User: 検索機能をいい感じにして

AI:
理解としては、検索結果の使いやすさを改善したいという依頼です。
ただし、現時点では次が不明です。

- 検索UIの改善か、検索ロジックの改善か
- 0件表示、ローディング、エラー表示のどれを対象にするか
- APIやDB検索条件を変更してよいか
- 完了条件を何にするか

まず今回は、UI上の検索結果表示だけを対象にし、APIやDBは変更しない方針でよいですか？
```

このように、AIはまず認識を揃えます。

## 記録する場所

要件認識合わせは、次のいずれかに残します。

- `templates/requirement-alignment.md`
- `templates/task-brief.md`
- Issue
- PR
- devlog

重要なのは、チャットの中だけに閉じ込めないことです。
後から見た人が、どの前提で実装したのか分かる状態にします。

## AGENTS.mdに入れるべきルール

プロジェクトの `AGENTS.md` には、次のようなルールを入れます。

```text
ユーザーの意図、作業範囲、制約、完了条件が曖昧な場合、実装に進んではいけない。

実装前に、AI側の理解、不明点、懸念点、置いた仮定を明示し、必要に応じてユーザー確認を行う。

プロダクトの挙動、UX、データ扱い、セキュリティ、コスト、公開範囲に影響する不明点を、AIが勝手に補完してはならない。
```

## この原則の位置づけ

Requirement Alignment Firstは、すべての品質ゲートより前にあります。

認識合わせができていない状態では、implementation plan、実装、テスト、レビューに進んでも意味がありません。
間違った前提のまま高品質に作っても、ユーザーにとっては間違ったプロダクトになるからです。
