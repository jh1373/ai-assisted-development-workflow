# Design Rationale

このドキュメントは、このワークフローをどのような問題意識と設計方針で構成しているかを説明します。

目的は、AI駆動開発を特別なプロンプト技術として扱うことではありません。
AIを使った開発を、追跡でき、検証でき、レビューでき、次の作業者が再開できる開発プロセスに近づけることです。

## 問題意識

AI駆動開発では、実装速度が上がります。
一方で、次の問題も起きやすくなります。

- 要件が曖昧なまま、AIがそれらしい実装を始める
- ユーザーの意図とAI側の理解がズレたまま作業が進む
- チャットに前提、判断理由、失敗した試行が埋もれる
- Gitには差分が残るが、なぜその変更にしたかが残らない
- AIが範囲外の変更や不要なリファクタリングを混ぜる
- コードが増えるほど、どのディレクトリが何を担当しているか分かりにくくなる
- AIにコード全体を読ませようとして、重要な文脈より実装詳細にコンテキストを使ってしまう
- テスト、ビルド、手動確認が曖昧なまま完了扱いになる
- 別チャットや別の作業者が再開するときに前回までの開発状況を把握できない
- 認証、権限、課金、個人情報、本番設定などをAI判断だけで変更してしまう

このワークフローは、AIの利用を制限するためのものではありません。
AIで実装速度を上げながら、要件、判断、検証、公開可否を人間が管理できる状態を作るためのものです。

## 設計方針

### 1. 実装前に認識を揃える

AIは、不足している情報を推測で補うことがあります。
その推測がプロダクトの挙動、UX、データ扱い、セキュリティ、コスト、公開範囲に影響する場合、実装に進むべきではありません。

そのため、Requirement Alignment Gateを最初に置きます。
ここでは、ユーザーの元要望、AI側の理解、対象範囲、対象外、完了条件、不明点、AIが置いた仮定を確認します。

詳細は [requirement-alignment.md](requirement-alignment.md) を参照してください。

### 2. 全体計画とタスク実行計画を分ける

プロジェクト全体の方向性と、今回の1タスクの実行計画を混ぜると、AIが大きすぎる作業を一度に進めやすくなります。

このワークフローでは、ROADMAPを全体の地図、PROJECT_STATUSを現在地、implementation planを今回の1タスクの実行計画として扱います。
session-startでは、ROADMAPとPROJECT_STATUSから今回取り組む候補タスクを確認し、実装前に細かな実行計画へ落とし込みます。

### 3. Directory Mapで読む範囲を絞る

大規模なプロジェクトでは、AIにコード全体を読ませることは非効率です。
コンテキストを大量に消費し、重要な責務境界より細かな実装詳細に引っ張られやすくなります。

このワークフローでは、`docs/DIRECTORY_MAP.md` を使い、主要ディレクトリ、責務、タスク別の参照先、触ってはいけない境界を先に確認します。
session-startでは必ずDIRECTORY_MAPを読み、今回のタスクで読むべき範囲を絞ります。
session-endではDIRECTORY_MAPの更新要否を確認しますが、更新するのは構造や責務に影響があった場合だけです。

### 4. 作業範囲と完了条件を先に固定する

AI駆動開発では、実装が速いぶん、作業範囲も広がりやすくなります。
そのため、実装前に implementation plan を作り、変更するもの、変更しないもの、検証方法、完了条件を固定します。

途中で方針を変えることは許容します。
ただし、なぜ変えたのかを completion review や devlog に残します。

### 5. チャットではなく追跡可能なファイルに残す

チャットは作業場としては有効ですが、正式な記録には向きません。
長くなりやすく、検索しにくく、次の作業者が前提を追いにくいためです。

このワークフローでは、判断理由と検証結果を次のように分けて残します。

- 現在地: `docs/PROJECT_STATUS.md`
- 主要ディレクトリと責務: `docs/DIRECTORY_MAP.md`
- タスク単位の認識合わせ、計画、完了レビュー: `docs/tasks/`
- 作業後の判断理由と検証結果: `docs/devlog/`
- 長期的な設計判断: `docs/adr/`

### 6. AIと人間の責任範囲を分ける

AIは調査、実装案、テスト案、ドキュメント下書き、エラー原因の仮説出しに使えます。
しかし、要件確定、リスク許容、公開判断、セキュリティ判断、最終レビューの責任は持てません。

このワークフローでは、AIの出力を成果物ではなくレビュー対象の提案として扱います。

詳細は [ai-human-boundary.md](ai-human-boundary.md) を参照してください。

### 7. 完了を実装済みではなく検証済みとして扱う

コードが生成されたことは完了ではありません。
完了とは、目的を満たし、検証され、残るリスクが記録され、次の作業者がチャット履歴なしで再開できる状態です。

そのため、completion review、verification gate、definition of doneを用意しています。

詳細は [definition-of-done.md](definition-of-done.md) と [quality-gates.md](quality-gates.md) を参照してください。

### 8. 手順の重さをタスクのリスクに合わせる

すべての作業に重い手順を強制すると、運用は続きません。
一方で、高リスクな変更を軽い確認で済ませると、AIによる変更の危険が大きくなります。

このワークフローでは、導入時にモードを固定せず、タスクごとにMinimal、Standard、Strictの3つのモードを使い分けます。
AIが今回のタスク内容とリスクを見て判定し、理由をimplementation planに記録します。

- Minimal: 小さく、戻しやすく、低リスクな変更
- Standard: 通常の機能追加、バグ修正、UI改善
- Strict: 認証、権限、課金、個人情報、本番設定、公開リリース、設計変更

詳細は [workflow-modes.md](workflow-modes.md) を参照してください。

## 標準的なタスク進行モデル

このワークフローでは、プロジェクト全体を整理する段階と、1つのタスクを実行する段階を分けます。

```text
作りたいものを決める
→ AIと壁打ちして目的、制約、成功条件を整理する
→ ROADMAPを作る
→ PROJECT_STATUSに現在地を残す
→ DIRECTORY_MAPに主要ディレクトリと責務を残す
→ session-startで現在の開発状況を把握する
→ ROADMAPとPROJECT_STATUSから今回のタスクを選ぶ
→ DIRECTORY_MAPで今回読むべき範囲を絞る
→ AIがタスク内容とリスクを見てWorkflow Modeを判定する
→ implementation planで今回の実行計画を作る
→ AIと実装する
→ テストと手動確認を行う
→ 完了前レビューを行う
→ devlogとGitに記録する
→ 必要ならPRでレビューする
```

1タスク内では、次の順番で進めます。

```text
ユーザーの要望またはROADMAP上の候補タスクを受け取る
→ AI側の理解を要約する
→ 不明点、曖昧な点、懸念点を洗い出す
→ 必要ならユーザーに確認する
→ 合意できた範囲だけをタスク化する
→ 現在の開発状況を把握する
→ DIRECTORY_MAPで関連ディレクトリと境界を確認する
→ Workflow Modeを判定する
→ 実装計画を作る
→ AIと実装する
→ テストと手動確認を行う
→ 完了前レビューを行う
→ devlogとGitに記録する
→ 必要ならPRでレビューする
```

実際の進め方は [practical-guide.md](practical-guide.md) を参照してください。

## 防止したい失敗

このワークフローは、特に次の失敗を防ぐために設計しています。

- 曖昧な要望のままAIが実装を始める
- AIが未確認の前提を勝手に補完する
- 目的と関係ない変更が混ざる
- ディレクトリ責務が曖昧になり、タスクごとに場当たり的な配置が増える
- コード全体を読もうとして、必要な文脈を絞れない
- テストや手動確認なしで完了扱いにする
- 失敗した試行や見送った案が消える
- セキュリティやプライバシーに関わる変更を軽く扱う
- 次の作業者がチャット履歴なしでは再開できない

具体的な失敗パターンは [anti-patterns.md](anti-patterns.md) を参照してください。

## 現時点の制約

このリポジトリを導入するだけで、すべてのプロジェクトが自動的に高品質になるわけではありません。
プロジェクト固有のテスト、ビルド、CI、レビュー体制、リリース判断、セキュリティ基準は別途必要です。

また、テンプレートを埋めること自体が目的になった場合、運用は重くなり、効果も下がります。
重要なのは、作業のリスクに見合った証拠を残すことです。

チームで使う場合は、誰が要件を確定し、誰がレビューし、誰が公開判断をするかを先に決める必要があります。
詳細は [team-development.md](team-development.md) を参照してください。
