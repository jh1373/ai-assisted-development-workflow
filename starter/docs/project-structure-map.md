# プロジェクト案内図

プロジェクト案内図は、ファイルを並べるだけの一覧ではありません。新規プロジェクトの「どの部分が何を担当し、処理がどこを通り、今回どこを見るべきか」を初心者向けの言葉で確認できる、閲覧専用のlocalhost画面です。

## 解決する問題

通常のファイル一覧から分かるのは名前と配置だけです。プロジェクト案内図では次を確認できます。

- どの役割がプロダクトのどの部分を担当するか
- 入口となるファイルはどれか
- ある処理がどの順番で進むか
- 今回のタスクでどこを見て、どこを触らないか
- ファイルの説明が古い、または不足していないか

ソースコード内容は走査、保存、配信しません。

## 5つの確認方法

### プロジェクト全体

プロジェクトを、ユーザー画面、機能、データ、運用基盤、開発手順、記録という役割に分けます。最初から全ファイルを見せず、プロジェクト全体の仕組みを表示します。

### どう動くか

ユーザー操作や開発手順がどのファイルを通るのか、順番に表示します。各段階から該当ファイルの役割説明へ移動できます。

### 作業場所を探す

作業内容ごとに、最初に見る場所、関連して確認する場所、今回触らない場所を分けます。

### ファイル構成

最初にプロジェクト直下の主要なフォルダと入口をカードで確認し、必要な場所だけツリーで開きます。各項目には役割説明と説明の根拠を表示します。

### 説明不足を確認

説明がない項目、プロジェクト固有の説明がないファイル、参照先が見つからない説明を表示します。

## 数字の読み方

画面上部では数字だけを表示せず、分母と判断結果を併記します。

- 役割を説明できる項目: `説明できる項目数 / 全項目数`と進捗線
- 個別説明があるファイル: 共通説明ではなく、そのファイル専用の説明を登録した件数
- 説明の見直し: 説明なし、個別説明不足、存在しない場所への案内の合計

問題がない場合は、次に使う「作業場所を探す」まで文章で案内します。確認事項がある場合は、「説明不足を確認」へ誘導します。

## 可読性の基準

初心者が拡大操作をしなくても読めることを前提にします。

- 標準本文は16px
- 補足情報も12px未満にしない
- サイドメニューと主要操作は15px以上
- 入力欄と主要ボタンは高さ44px以上
- 文字色は薄くしすぎず、背景との区別を保つ
- スマートフォンでは数値を1列にして、説明を省略しない

## ファイルの役割説明

入口、中心ファイル、重要ファイルには、個別の役割説明を登録します。

主な項目:

- path、display_name、beginner_summary
- responsibility、not_responsible_for
- when_to_change、inputs、outputs
- depends_on、area、layer、importance、evidence
- boundaries、task_types

AIはファイルを作成または改名した同じタスク内で役割説明を更新します。後からファイル名やコード内容だけを見て、説明を推測で確定しません。

補助ファイルは、正確であれば親フォルダの説明を引き継げます。個別の説明と引き継いだ説明は画面上で区別します。

## schema version 2

.ai-workflow/directory-map.jsonが意味情報の正本です。

| 設定項目 | 保存する内容 |
|---|---|
| areas | プロジェクトを主な役割と分類に分ける |
| nodes | ファイルとフォルダの役割説明 |
| flows | 処理や開発手順の順序 |
| task_lenses | 作業内容ごとの確認範囲 |
| conventions | ほかに説明がない場合の補助規則 |

schema version 1も読み取れますが、主な役割、処理の流れ、作業場所の案内は表示されません。新規プロジェクトはversion 2を使用します。

## 役割の根拠

優先順位:

1. 個別に登録した役割説明
2. 登録されたパターン
3. 最も近い親区域・ディレクトリ
4. 一般ファイル名・拡張子
5. 説明なし

一般的な「Markdownファイル」より、親区域の「プロジェクト計画を保存する」の方を優先します。

## 構造と意味情報

構造と説明は別に評価します。

- 構成の一致: 確認済みの場所と種類が、現在の構成と一致するか
- 役割説明の有無: 全項目を根拠付きで説明できるか
- 個別説明の有無: ファイル単位の説明が登録されているか
- 参照先の確認: 役割説明、処理の流れ、作業場所の案内が存在しない場所を指していないか

初回設定中は、作成予定の場所がまだ存在しなくても構いません。JSONのpatternで計画済みなら、壊れた参照として扱いません。

## 初回設定

コード作成前に次を設計します。

1. 役割区域と層
2. 各区域の初心者向け説明
3. 入口となる予定ファイル
4. 主要な処理や開発手順の流れ
5. 最初の作業で見る場所
6. 触ってはいけない境界

初期状態の機械判定は`DIRECTORY_MAP_PROVISIONAL`です。実際の構成と役割説明をユーザーが確認した後だけ、確認済みの状態へ進めます。

## 通常タスク

タスク開始時は、作業場所の案内を全ファイル一覧より先に確認します。対応する案内がなければ、追加候補としてimplementation planへ記録します。

入口または中心ファイルを作成、改名、役割変更した場合、同じ変更内で役割説明を更新します。

タスク終了時は構成の変更だけでなく、主な役割、ファイルの役割説明、処理の流れ、作業場所の案内への影響を確認します。

## コマンド

Windowsではプロジェクト直下のopen-project-structure-map.cmdをダブルクリックします。

    python scripts/project-structure.py validate
    python scripts/project-structure.py diff
    python scripts/project-structure.py generate
    python scripts/project-structure.py serve --open-browser

初回承認:

    python scripts/project-structure.py verify --verified-by "User"

承認済み変更の基準線更新:

    python scripts/project-structure.py refresh --by "User"

## 固定出力

| 機械判定 | 意味 |
|---|---|
| DIRECTORY_MAP_PROVISIONAL | 初回構築と意味情報の確認中 |
| DIRECTORY_MAP_VERIFIED | 承認済み構造と一致 |
| DIRECTORY_MAP_DRIFT_DETECTED | 承認後の構造差分あり |
| DIRECTORY_MAP_INVALID | JSON、snapshot、意味情報の形式が不正 |
| DIRECTORY_MAP_CHECK_FAILED | Python、権限、読み取りなどで検査不能 |

## セキュリティ

- 127.0.0.1だけで待ち受ける
- 起動ごとに一時tokenを生成する
- APIはtokenなしアクセスを拒否する
- リポジトリ全体を静的配信しない
- ソースコード内容を読まない、保存しない、配信しない
- symlink先を走査しない
- 上位パスへ出る定義を拒否する
- Content Security Policyを付ける
- 画面からJSONや基準線を書き換えない

## Python

Python 3.10以降を推奨します。外部Pythonパッケージは使用しません。

Pythonがない場合はDIRECTORY_MAP_CHECK_FAILEDとして扱います。ランチャーもツールを自動インストールしません。
