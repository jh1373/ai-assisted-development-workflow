# Practical Guide

このドキュメントは、AI駆動開発で1つのタスクをどう進めるかを説明します。

テンプレートを埋めることが目的ではありません。
目的は、AIを使っても作業範囲、判断理由、検証結果、残課題が追える状態にすることです。

## 1. 要件認識合わせを行う

AIに実装を頼む前に、まずタスクを明確にします。

確認すること:

- ユーザーの元要望は何か
- AI側はどう理解したか
- 何を達成したいか
- なぜ今やるのか
- どの画面、API、ファイル、データが関係しそうか
- 何は今回やらないのか
- 何ができたら完了と言えるか
- どう検証するか
- AIが勝手に置いた仮定はないか
- ユーザーに確認すべき不明点はないか

曖昧なままAIに渡すと、AIは足りない情報を推測します。
その推測が外れると、見た目は進んでいても、違うものを作っている状態になります。

必要なら `templates/requirement-alignment.md` と `templates/task-brief.md` を使います。
未確認の前提が残る場合は、実装に進みません。

## 2. 現在地を復元する

新しいチャットや新しいAIツールで作業を始めるときは、最初に現在地を復元します。

確認すること:

- `git status --short --branch`
- `README.md`
- `AGENTS.md`
- `docs/PROJECT_STATUS.md`
- `docs/ROADMAP.md`
- 直近の `docs/devlog/`
- 最後に通ったテストやビルド

ここを省くと、AIは古い前提や不足した文脈で作業します。
結果として、既存設計に合わない実装や、前回見送った方針を再実行する危険が増えます。

手順は `workflows/session-start.md` を使います。

## 3. 実装計画を作る

実装前に、計画を短く書きます。

計画で固定すること:

- 目的
- 背景
- 変更するもの
- 変更しないもの
- 実装方針
- 影響範囲
- リスク
- 検証方法
- 完了条件

計画は長文である必要はありません。
重要なのは、AIが勝手に範囲を広げたときに「それは今回の対象外」と判断できる基準を持つことです。

テンプレートは `templates/implementation-plan.md` を使います。

## 4. AIに小さく任せる

AIには大きなタスクを一度に投げない方が安全です。

良い任せ方:

- 関連ファイルを調べる
- 実装案を比較する
- 小さな差分を作る
- テストケースを洗い出す
- エラー原因の候補を出す
- ドキュメントの下書きを作る

危険な任せ方:

- 仕様が曖昧なまま実装させる
- ユーザーに確認すべき不明点をAIが勝手に補完する
- 大量のファイルを一括で変更させる
- 認証、権限、課金、個人情報をAI判断だけで変える
- テストせずに完了扱いにする
- AIの説明だけを根拠にマージする

AIの出力は、完成品ではなくレビュー対象の提案です。

## 5. 差分を確認する

実装後は、コードが動くかだけでなく、差分が計画に合っているかを確認します。

確認すること:

- 計画した範囲だけが変わっているか
- 意図していないファイルが変わっていないか
- 既存の設計や命名に沿っているか
- 不要なリファクタリングが混ざっていないか
- エラー処理、境界値、権限、データ扱いに問題がないか

予定外の変更が必要になった場合は、隠さず記録します。

## 6. 検証する

完了判断は、気分ではなく証拠で行います。

例:

```bash
npm test
npm run build
git diff --check
```

UIや操作が関係する場合は、手動確認も記録します。

記録する内容:

- 実行したコマンド
- 結果
- 失敗した場合の原因
- スキップしたチェックと理由

「AIが正しいと言った」は検証ではありません。

## 7. 完了前レビューを行う

完了前に、計画と実際の結果を照合します。

確認すること:

- 目的は達成されたか
- 完了条件を満たしたか
- 未完了のことは何か
- 計画から変わったことは何か
- 検証結果は十分か
- 次の作業者が再開できるか

テンプレートは `templates/completion-review.md` を使います。
完了判断の基準は `docs/definition-of-done.md` を使います。

## 8. 判断理由を残す

最後にdevlogを書きます。

devlogに残すこと:

- なぜこの作業をしたか
- 何を変更したか
- どの方針を採用したか
- どの選択肢を見送ったか
- 何を検証したか
- 何が未完了か
- 次に何を見るべきか

devlogは作業日報ではありません。
将来の自分や他の開発者が、判断を追うための記録です。

テンプレートは `templates/devlog.md` を使います。

## 9. 重要な設計判断はADRに分ける

devlogはセッション単位の記録です。
長期的に残すべき設計判断はADRに分けます。

ADRに残すべき例:

- 認証方式を変える
- 状態管理ライブラリを採用する
- DB構造を変える
- 外部サービスを導入する
- アーキテクチャを変更する

テンプレートは `templates/adr.md` を使います。

## 10. PRとCIに接続する

GitHubで開発する場合は、PRに次を残します。

- 何を変えたか
- なぜ変えたか
- どのモードで作業したか
- 関連する計画、レビュー、devlog、ADR
- 実行した検証
- 残るリスク

`.github/pull_request_template.md` を使うと、レビュー時に確認しやすくなります。

CIでは `scripts/check-docs.sh` と `.github/workflows/docs-check.yml` を使い、最低限の文書構造、リンク切れ、秘密情報らしき文字列を確認します。

## まとめ

AI駆動開発で重要なのは、AIを使うこと自体ではありません。

重要なのは、AIで速くなった作業を、人間が検証できる形に戻すことです。

そのために、このガイドでは次を残します。

- 目的
- 範囲
- 判断理由
- 検証結果
- 未完了事項
- 次の再開地点
