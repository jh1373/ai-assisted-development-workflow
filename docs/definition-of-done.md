# Definition of Done

このドキュメントは、このワークフローにおける「完了」の基準を定義します。

AI駆動開発では、実装が速く終わったように見えることがあります。
しかし、完了判断は気分やAIの説明ではなく、証拠で行います。

## 共通の完了条件

タスクは、次を満たしたときにだけ完了です。

- 元の目的を満たしている
- 最終的な作業範囲が明確である
- 計画した変更が完了している
- 予定外の変更がある場合、その理由が説明されている
- 必要なテスト、ビルド、手動確認を実行している
- 失敗したチェックやスキップしたチェックが記録されている
- 必要なドキュメントが更新されている、または不要と判断した理由がある
- セキュリティとプライバシーへの影響を確認している
- 残るリスクが記録されている
- 次のセッションがチャット履歴なしで再開できる

不明な項目がある場合、そのタスクは完了ではありません。

## 強い証拠と弱い証拠

完了を主張するなら、具体的な証拠を残します。

強い証拠:

- 実行したコマンドとpass/failの要約
- 追加または変更したテスト
- 手動確認の手順と観察結果
- UI変更のスクリーンショットや録画
- 関連するIssue、PR、ADR
- スキップしたチェックと理由

弱い証拠:

- 見た感じ問題なさそう
- AIが正しいと言っている
- エラーは出ていないと思う
- 小さい変更なので大丈夫
- たぶん動く

弱い証拠だけでは完了にしません。

## Minimal Modeの完了条件

小さく低リスクな作業では、Minimal modeで十分です。

完了条件:

- 目的と完了条件を書いた
- 実装がその目的に合っている
- 最低1つの意味ある検証を行った
- 将来の作業者に役立つメモを残した

## Standard Modeの完了条件

通常の機能追加やバグ修正では、Standard modeを基本にします。

完了条件:

- `implementation-plan.md` または同等の計画がある
- 計画と実際の結果を照合している
- テストまたは手動確認を実行している
- `completion-review.md` または同等の完了前レビューがある
- `devlog.md` または同等の判断ログがある
- プロジェクト状態が変わった場合、`PROJECT_STATUS.md` を更新している

## Strict Modeの完了条件

高リスクな変更では、Strict modeを使います。

Standard modeの条件に加えて、次が必要です。

- 明示的なリスクレビュー
- セキュリティとプライバシーの確認
- ロールバックまたは復旧方法
- 責任を持つ人間のレビュー
- リリースやデプロイへの影響記録
- 未解決のブロッカーを隠していないこと

Strict modeが必要な領域:

- 認証
- 権限
- 課金
- 個人情報
- DBマイグレーション
- 外部連携
- 依存ライブラリ追加
- 本番設定
- 公開リリース

## 完了ではない例

次の状態は完了ではありません。

- AIがコードを生成したが、テストを実行していない
- テストは通ったが、作業範囲が途中で変わった理由がない
- バグは直ったように見えるが、再現条件が記録されていない
- UI変更なのに画面確認をしていない
- セキュリティに関わる変更なのにリスクレビューがない
- 未コミットの変更が残ったままセッションを終えている
- 判断理由がチャットにしか残っていない

完了とは、作業が終わった気がする状態ではありません。
次の人が安全に理解し、検証し、必要なら戻せる状態です。
