# AIと人間の役割分担

AI開発支援ツールを使う場合でも、すべてをAIに任せるわけではありません。
このワークフローでは、AIに任せる作業と、人間が判断する作業を分けます。

## Project Initializationでの役割

AIに任せること:

- 壁打ちの質問を段階的に提示する
- ユーザーの回答を要約する
- Confirmed、Hypothesis、Unknown、Deferredの候補を整理する
- PROJECT_BRIEF、ROADMAP、AGENTS.mdの下書きを作る
- 文書間の矛盾と未確認事項を指摘する
- 初期設定チェッカーを実行する

人間が判断すること:

- プロダクトの目的と対象ユーザーをどう扱うか
- どの仮説を最初に検証するか
- どのリスクと不明点を残したまま開始するか
- ROADMAPとプロジェクト固有ルールを受け入れるか
- 初期設定を`ready`にしてよいか

AIは初期設定文書を下書きできますが、ユーザー承認を代行できません。
初期設定状態も文章から推測せず、判定スクリプトの固定結果を使います。

## Project Structure Mapでの役割

AIに任せること:

- 全パスの機械走査と構造差分の提示
- 一般的なファイル名や親ディレクトリから説明できる役割の表示
- JSON正本、snapshot、生成Markdownの整合性確認
- 新しい役割、境界、タスク参照先の候補提示

人間が判断すること:

- 主要ディレクトリと重要ファイルの正式な責務
- AIが提案した役割と境界を受け入れるか
- ProvisionalからVerifiedへ変更してよいか
- 構造差分を新しい基準線として受け入れるか

AIは、検証を通すためだけに未分類項目へ推測した役割を設定しません。

## AIに任せやすい作業

- 既存コードの読解補助
- 実装案の比較
- 小さな実装のたたき台作成
- テストケースの洗い出し
- エラー原因の候補出し
- ドキュメントの下書き
- リファクタリング候補の提示

## 人間が判断する作業

- ユーザーの要望をどう解釈するか
- 何を作るか
- 何を作らないか
- ユーザーにとって必要か
- 既存仕様を変えてよいか
- セキュリティ上問題がないか
- テスト結果をもとに完了と判断してよいか
- 公開してよい情報か

## 注意すべきこと

AIの回答は、判断材料の1つです。
最終判断は、コード、仕様、テスト結果、運用条件を確認したうえで行います。

特に次の変更は、AIの提案をそのまま採用しません。

- アーキテクチャ変更
- ライブラリ追加
- 認証や権限に関わる変更
- データ構造の変更
- 既存動作を変える変更
- 外部公開に関わる変更
