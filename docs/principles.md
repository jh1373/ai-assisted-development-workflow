# Principles

AI駆動開発は、エンジニアリング判断を省略するための近道ではありません。
実装速度を上げながら、判断、証拠、責任を見える状態に保つための開発方法です。

このワークフローは、次の原則を前提にしています。

## 1. チャットは作業場であり、信頼できる記録ではない

AIとのチャットは、調査、実装補助、レビュー、発想出しには有効です。
しかし、プロジェクトの正式な記録には向いていません。

正式な記録は、レビューでき、Gitで管理でき、他の人が読めるファイルに残します。

- 要件と作業範囲は task brief
- 実装意図は implementation plan
- 重要な設計判断は ADR
- 検証結果は completion review
- 現在地は `PROJECT_STATUS.md`
- 判断理由は devlog

将来の開発者が古いチャットを読まないと理解できないなら、その運用は失敗です。

## 2. Gitは何を変えたかを残すが、なぜ変えたかは残さない

Gitの差分を見ると、どのファイルがどう変わったかは分かります。
しかし、次のことは十分には分かりません。

- なぜその変更が必要だったのか
- どの選択肢を見送ったのか
- どのリスクを受け入れたのか
- 何を検証したのか
- 何が未完了なのか

AI駆動開発では、実装速度が速いぶん、理由が失われる速度も速くなります。
そのため、差分とは別に判断理由を残します。

## 3. 実装前に範囲を固定する

AIは、曖昧な依頼でもそれらしい実装を返します。
これは便利ですが、危険でもあります。

実装前に必ず決めます。

- 解決する問題
- 変更する範囲
- 変更しない範囲
- 完了条件
- 検証方法
- リスク

途中で範囲を変えても構いません。
ただし、なぜ変えたのかを記録します。

## 4. AIの出力は提案であり、権威ではない

AIはコード、テスト、設計、文章を提案できます。
しかし、それを採用してよいかを判断する責任は人間にあります。

特に次の変更は、AIの提案をそのまま採用しません。

- 認証
- 権限
- 課金
- 個人情報
- データ移行
- アーキテクチャ変更
- 依存ライブラリ追加
- 本番公開

AIは作業者を補助します。
責任者にはなれません。

## 5. 完了とは、検証済みで再開可能な状態である

コードが書けたことは、完了ではありません。

完了とは、少なくとも次を説明できる状態です。

- 何を変えたか
- なぜ変えたか
- どう検証したか
- 何をやらなかったか
- 何が残っているか
- 次の作業者がどこから再開すべきか

完了判断の詳細は [definition-of-done.md](definition-of-done.md) を参照してください。

## 6. 不確実性を隠さない

AI駆動開発で危険なのは、分からないことを分かったように扱うことです。

次の情報は隠さず残します。

- 未確定の仕様
- 失敗した試行
- スキップした検証
- 採用しなかった案
- 残るリスク
- 人間の確認が必要な点

きれいな成功ストーリーに見せることより、次の判断に使える証拠を残すことを優先します。

## 7. 手順の重さはリスクに合わせる

すべての作業に重い手順を強制する必要はありません。

- 小さく戻しやすい変更は Minimal
- 通常の機能追加やバグ修正は Standard
- 認証、権限、課金、個人情報、本番影響は Strict

目的は文書を増やすことではありません。
目的は、曖昧な判断を減らすことです。
