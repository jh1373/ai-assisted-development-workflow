# Strict Mode

Strict modeは、失敗したときにユーザー、データ、収益、セキュリティ、長期保守性へ影響が出る作業に使います。

Standard modeを少し重くしたものではありません。
必要な証拠のレベルが違う運用です。

## Strict Modeが必要な作業

次を扱う場合はStrict modeにします。

- 認証
- 権限
- 課金、決済、サブスクリプション
- 個人情報
- データ移行
- 本番設定
- インフラ
- 公開リリース
- 依存ライブラリ追加または大きなアップグレード
- アーキテクチャ変更
- 取り消しにくい操作

迷う場合は、リスクが理解できるまでStrict modeとして扱います。

## 必須成果物

Strict modeでは、次を残します。

- task brief
- implementation plan
- security and privacy review
- rollback or recovery notes
- completion review
- devlog
- 大きな設計判断がある場合はADR

推奨テンプレート:

- `templates/security-review.md`
- `templates/rollback-plan.md`
- `templates/adr.md`

長文である必要はありません。
ただし、判断に使える証拠が必要です。

## セキュリティとプライバシー確認

完了前に、次を確認します。

- secrets、tokens、cookies、credentialsに触れているか
- 誰が何にアクセスできるかが変わるか
- 個人情報や顧客情報を扱うか
- ログに秘密情報や個人情報が出る可能性があるか
- 新しい外部依存や外部連携を追加するか
- デプロイ、ホスティング、実行環境の設定を変えるか
- データ移行や削除が必要か

該当する場合は、影響と対策を記録します。

## ロールバックと復旧

高リスク変更では、次を記録します。

- どう無効化、差し戻し、緩和できるか
- データを戻せるか
- ロールバック後に何を確認すればよいか
- 誰がロールバック判断をするか

ロールバックできない場合は、できないことを明記します。
その場合、リスクを人間が承認する必要があります。

## 人間の承認

AIは証拠を準備できます。
しかし、Strict modeの最終承認者にはなれません。

責任者は次を確認します。

- その挙動は意図したものか
- リスクは許容できるか
- 検証証拠は十分か
- 未解決事項を理解しているか
- リリースしてよいか

## 完了ゲート

Strict modeのタスクは、次の状態では完了ではありません。

- リスクレビューがない
- 検証が一般論だけで書かれている
- スキップしたチェックの理由がない
- ロールバックが検討されていない
- ログやドキュメントに秘密情報が含まれる可能性がある
- 最終状態をチャット履歴なしで理解できない
- 人間の承認が必要なのに記録されていない

Strict modeの目的は、慎重そうに見せることではありません。
問題が起きたときに、何を判断し、何を検証し、どう戻すかを追える状態にすることです。
