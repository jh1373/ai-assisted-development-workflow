# ADR Guidelines

ADRは、将来の開発者がチャット履歴を読まなくても重要な設計判断を理解できるようにするための記録です。

devlogは1タスクごとの作業記録です。
ADRは、その中でもプロジェクトの設計、運用、保守に長く影響する判断だけを切り出します。

## 基本ルール

- AIはADRを勝手に確定しない
- AIはADR候補を検知したら、ユーザーに確認する
- ユーザーが了承した場合のみ、`templates/adr.md` を使ってADRを作成する
- ADRを作成しない場合でも、devlogのADR欄に「不要」と理由を書く

## ADR候補として検知する条件

次のいずれかに当てはまる場合、AIはADR候補として扱います。

- 認証、認可、権限管理を変更する
- DB構造、データ保存方式、データ移行に関わる
- 状態管理、ルーティング、API方式など基盤設計を変える
- 新しい外部サービス、ライブラリ、SaaSを導入する
- ディレクトリ構成や責務分離を大きく変える
- パフォーマンス、セキュリティ、運用コストに長期影響がある
- 複数の選択肢から1つを選び、後戻りコストが高い
- チームや将来の自分が「なぜこれにしたのか」を知る必要がある

## AIがユーザーに確認すること

ADR候補を検知した場合、AIは実装前または完了前レビューで次を確認します。

```text
この変更は長期的な設計判断に影響するため、ADR候補です。

- 判断内容:
- 影響範囲:
- 見送った選択肢:
- 後戻りコスト:

ADRとして docs/adr/ に記録しますか？
```

ユーザーが了承した場合はADRを作成します。
了承がない場合は、devlogに「ADR候補だったが、今回は作成しない」と理由を残します。

## ADRを作るタイミング

ADRは、判断をした直後に作るのが理想です。

実務では次のどちらかで作成します。

- 実装前: 設計方針を先に固定する必要がある場合
- 完了前レビュー時: 作業中に重要判断が発生した場合

タスク完了後に思い出して書くと、判断理由が曖昧になりやすいため避けます。

## ADRにしなくてよい例

- 文言を変更する
- UIの余白や色を調整する
- 小さなバグを修正する
- 既存方針に沿って処理を追加する
- 一時的な検証コードを書く
- devlogに理由を書けば十分な小さな作業判断

## devlogとの関係

devlogには、ADRを作ったかどうかを必ず書きます。

例:

```text
ADR:
- docs/adr/0001-use-server-side-session-storage.md
```

ADRが不要な場合:

```text
ADR:
- 不要。既存方針に沿ったUI改善であり、長期的な設計判断ではないため。
```
